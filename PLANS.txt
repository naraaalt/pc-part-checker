STATUS: UNDECIDED — blocked on deployment target choice.
The app is currently a pure Vite + React 19 + TS static SPA (no server code).
A backend is ONLY needed to upgrade Share/Permalink (#4) from a self-contained
URL to a short-link (see its BACKEND COMPATIBILITY notes). It is NOT required
for any other feature today.

Constraints / decision drivers:
- Must run on FREE, trusted hosts: leaning Vercel and/or GitHub Pages.
- GitHub Pages CANNOT run server code -> backend must be SERVERLESS (functions)
  to stay portable across these hosts.
- Backend scope (decided): stores ONLY share payloads; builds stay in localStorage.
- Short-link route already reserved: /s/:id.

Candidate approaches (evaluating):
1. Serverless functions + managed KV/store
   - e.g. Vercel Functions + Vercel KV / Neon Postgres; Netlify Functions + Blob.
   - Best fit for Vercel + free tier; no always-on server.
2. BaaS (Supabase / Firebase)
   - Auth + DB + storage in one; more than needed for share-only, but easy.
   - Supabase has a free tier and works with any static host.
3. Small full server (Node/Express on Render / Railway)
   - Simple but needs an always-on host -> conflicts with GitHub Pages-only plan.

Open questions before committing:
- Where will the app actually be deployed? (Vercel vs GitHub Pages vs both)
- Does share-short-link even matter enough to justify a backend, or is the
  self-contained URL (already designed, zero backend) good enough?
- Any future features that would REQUIRE a backend (accounts, cloud sync)?
